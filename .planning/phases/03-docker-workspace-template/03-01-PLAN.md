## Phase Goal (MVP user story)

**As a** developer using this Coder deployment, **I want to** create a workspace from the Docker template and get a working browser VS Code session and a JetBrains Gateway (IntelliJ IDEA) connection with a home directory that survives stop/start, **so that** I can do real development without losing my work between sessions.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Coder server → host Docker daemon | Coder server provisions workspace containers via the mounted `/var/run/docker.sock`; full host-equivalent privilege lives here |
| Workspace container → Coder server | Agent dials back over `host.docker.internal:7080` carrying `CODER_AGENT_TOKEN` |
| Template author (admin) → template | `workspace_image` and `docker_socket` are template-level inputs set by admins, not end-users (D-03: no `coder_parameter`) |
| Developer browser/Gateway → workspace apps | code-server (`--auth none`) and Gateway apps are reached through the Coder proxy auth layer |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | Host Docker socket | accept | Only the Coder server container mounts `/var/run/docker.sock` (compose.yaml). The workspace container created by this template does NOT mount the socket — no Docker socket volume/bind appears in `docker_container.workspace`. Acceptance criterion: socket is absent from the container resource. Severity controlled to low by design. |
| T-03-02 | Information Disclosure | `CODER_AGENT_TOKEN` | mitigate | Token passed via container `env` (process env), never written to a file or baked into the image; Coder rotates it on workspace restart. Template introduces no token persistence. |
| T-03-03 | Tampering | `workspace_image` variable | mitigate | Trust boundary documented: image var is template-level (admin-only; D-03 forbids `coder_parameter`), pinned to `codercom/enterprise-base:ubuntu`, fetched via `docker_image.main` with `keep_locally = true`. No user-supplied image path. |
| T-03-04 | Spoofing/Authentication Bypass | code-server `--auth none` | accept | Standard Coder pattern — the code-server module sets `--auth none` because the Coder proxy enforces authn/authz before requests reach the app. No bypass introduced by this template. |
| T-03-05 | Denial of Service | Unbounded workspace resource use (no CPU/mem limits) | accept | QOL-03 (resource limits) is explicitly deferred to v2; single-host scaffold; acceptable for MVP. Documented as deferred in REQUIREMENTS.md. |
| T-03-SC | Tampering | Coder registry modules / TF providers (supply chain) | mitigate | All modules/providers pinned exactly (`code-server 1.5.0`, `jetbrains-gateway 1.2.6`, `coder/coder ~> 2.18`, `kreuzwerker/docker ~> 4.4`) and approved in RESEARCH.md §"Package Legitimacy Audit" (all OK, first-party Coder registry + canonical TF providers). `hashicorp/docker` (archived) is banned. No `[ASSUMED]`/`[SUS]` packages → no blocking human checkpoint required. |

No HIGH-severity threat is left unmitigated. T-03-01 (the headline Docker-socket EoP) is structurally mitigated by NOT granting the workspace container socket access — enforced as an acceptance criterion in Task 1's verify.
</threat_model>
