# Phase 3: Code Review Report

**Reviewed:** 2026-06-17
**Depth:** standard
**Files Reviewed:** 2
**Status:** issues_found

## Summary

Reviewed the new Coder Docker workspace Terraform template (`templates/docker/main.tf`)
and the appended `## Workspace Template` section of `README.md`. Scope was limited to the
phase-3 changes per the scope note.

The template is largely correct and the high-risk pitfalls are handled well: `coder_agent`
has no `count` (token/init_script always generated), the home `docker_volume` is keyed on the
immutable workspace **UUID** (not the mutable name) and carries `lifecycle { ignore_changes }`,
`docker_image.main` uses `keep_locally = true`, and the `host.docker.internal` / `host-gateway`
wiring plus the `replace()` over the init script are present and paired. Provider and module
pins match CLAUDE.md (`coder/coder ~> 2.18`, `kreuzwerker/docker ~> 4.4`, code-server `1.5.0`,
jetbrains-gateway `1.2.6`). The intentional omission of the `claude-code`/`coder_ai_task`
AI-agent wiring is confirmed as deferred to v2 by the phase plan/context — **not** flagged.

No BLOCKER-class defects (no shell injection, no data-loss path, no socket exposure into
workspaces). The findings below are robustness, correctness-of-documentation, and quality
issues that should be addressed.

## Warnings

### WR-01: `replace()` uses Terraform regex syntax but pattern can over-match the agent token / URL path

**File:** `templates/docker/main.tf:197`
**Issue:** The entrypoint rewrites the agent init script with:
```hcl
replace(coder_agent.main.init_script, "/localhost|127\\.0\\.0\\.1/", "host.docker.internal")
```
Because the second argument is wrapped in `/.../`, Terraform treats it as a regex and replaces
**every** occurrence of `localhost` or `127.0.0.1` anywhere in the rendered init script — not just
the host component of `CODER_ACCESS_URL`. The init script is generated by the Coder server and may
contain the access URL more than once (e.g. in env exports and in the agent's `--url` flag). A blanket
substitution is the documented intent for the `127.0.0.1` quickstart, but it is brittle: if a future
server version embeds `127.0.0.1` in a context where it should stay literal (a comment, a health-probe
loopback, a bind address), this silently corrupts it. There is no validation that the substitution
only touched the URL.
**Fix:** Constrain the match to the URL scheme/host boundary, e.g. anchor on the `http://` prefix so
only the access-URL host is rewritten:
```hcl
entrypoint = ["sh", "-c", replace(
  coder_agent.main.init_script,
  "/(https?://)(localhost|127\\.0\\.0\\.1)/",
  "$${1}host.docker.internal"
)]
```
At minimum, document that the substitution is global and assumes `127.0.0.1`/`localhost` never appears
in the init script except as the access-URL host.

### WR-02: `lifecycle { ignore_changes = all }` on the home volume freezes the volume's labels and silently masks drift

**File:** `templates/docker/main.tf:124-126`
**Issue:** `ignore_changes = all` was chosen to protect the volume from destroy-on-rename, but `all`
is broader than needed. It causes Terraform to ignore **every** attribute including the `coder.owner`,
`coder.owner_id`, and `coder.workspace_name_at_creation` labels. If a workspace is transferred to a new
owner, the volume labels become permanently stale and misleading for any operator querying volumes by
owner label — and Terraform will never reconcile them. More importantly, `ignore_changes` does **not**
prevent destruction (only attribute updates), so the comment "Prevent Terraform from destroying the home
volume on workspace rename" is partially inaccurate: protection against rename-destroy comes from keying
`name` on the immutable UUID (line 120), not from this lifecycle block. The block is doing less than the
comment claims.
**Fix:** Scope `ignore_changes` to the attributes that actually need protection (e.g. `name`, or the
labels that legitimately reflect creation-time state), and correct the comment to attribute the
rename-safety to the UUID-based name:
```hcl
lifecycle {
  ignore_changes = [name]
}
```
If the intent is truly "never touch this volume again," keep `all` but fix the comment to state that
the destroy-protection comes from the UUID name, not from `ignore_changes`.

### WR-03: README "same-major shortcut" copies old PGDATA into a possibly-initialized volume

**File:** `README.md:189-200`
**Issue:** The raw-file-copy migration runs `cp -a /from/. /to/` from `coder_coder_data` into
`coder_coder_pgdata` and only afterwards runs `docker compose up -d`. If the operator has **already**
run `docker compose up -d` once on the new scaffold (a very common state — the Quick Start section tells
them to do exactly that), `coder_coder_pgdata` already contains an initialized postgres:17 cluster. The
`cp -a /from/. /to/` then overlays the old cluster files **on top of** the new ones without clearing the
target, producing a mixed/corrupt data directory (stale `postmaster.pid`, mismatched `pg_control` vs
`base/`). The instructions never tell the reader to ensure the target volume is empty first.
**Fix:** Either instruct the operator to recreate/empty the target volume before copying, or make the
copy refuse a non-empty target:
```bash
# Ensure the target is empty first (DESTROYS any existing new-stack data):
docker volume rm coder_coder_pgdata 2>/dev/null || true
docker volume create coder_coder_pgdata
docker run --rm -v coder_coder_data:/from -v coder_coder_pgdata:/to \
  alpine sh -c 'cp -a /from/. /to/'
```
Add a warning that this shortcut must run against a **fresh** target volume.

### WR-04: README implies a stop-the-source-stack step is unnecessary for the raw copy migration

**File:** `README.md:189-200`
**Issue:** The same-major raw-copy path copies live PGDATA files but never instructs the operator to
**stop the old stack first**. Copying a Postgres data directory while the source `postgres` is running
yields an inconsistent snapshot (in-flight WAL, dirty pages not flushed), which can produce a cluster
that fails recovery or starts with silent corruption. The logical `pg_dump` path above it is online-safe;
the raw file copy is not, and the docs do not distinguish them.
**Fix:** Prepend a stop step to the raw-copy block:
```bash
# Stop the OLD quickstart stack so PGDATA is quiesced before copying:
docker compose -f /path/to/old/compose.yaml down
```
and note that, unlike `pg_dump`, the raw copy requires the source database to be stopped.

## Info

### IN-01: Metadata key numbering skips `2_`

**File:** `templates/docker/main.tf:90,98,106`
**Issue:** The `coder_agent` metadata keys are `0_cpu_usage`, `1_ram_usage`, `3_home_disk` — the `2_`
slot is skipped. Harmless (keys only need to be unique and sort), but it suggests a metadata block was
removed without renumbering, which can confuse future maintainers.
**Fix:** Renumber the disk key to `2_home_disk` for a contiguous sequence.

### IN-02: `lower()` applied to workspace name but not owner name in container name

**File:** `templates/docker/main.tf:191`
**Issue:** `name = "coder-${data.coder_workspace_owner.me.name}-${lower(data.coder_workspace.me.name)}"`
defensively lowercases the workspace name (Docker container names are case-sensitive but conventionally
lowercase) yet leaves `owner.name` un-normalized. Coder usernames are already lowercase-constrained today,
so this is not a live bug, but the asymmetry is inconsistent — if either segment ever contained an
uppercase character the name would still be valid for Docker but the intent of the `lower()` is undermined.
**Fix:** Apply `lower()` consistently to both interpolated segments, or drop it and rely on Coder's
username/workspace-name validation, documenting the assumption.

### IN-03: Default workspace image tag is mutable (`:ubuntu`)

**File:** `templates/docker/main.tf:42`
**Issue:** The default `workspace_image = "codercom/enterprise-base:ubuntu"` is a moving tag. Combined
with `keep_locally = true`, the first pull is cached and subsequent template applies will **not** pick up
upstream changes — but a fresh host will pull whatever `:ubuntu` points to that day, undermining the
reproducibility goal CLAUDE.md states for the Coder image itself. The inline comment already suggests
pinning to a digest; the default does not.
**Fix:** This is intentionally left to the operator and is documented, so no change required for v1.
Consider pinning the committed default to a `@sha256:` digest to match the project's reproducibility
stance, or add a README note that the default tag is unpinned by design.

### IN-04: README "extra_hosts" terminology does not match the template

**File:** `README.md:368-371`
**Issue:** The connectivity section says the template adds `host.docker.internal` "via an `extra_hosts` /
`host-gateway` entry." The `kreuzwerker/docker` provider uses a `host { host = ...; ip = ... }` block
(main.tf:204-207), not an `extra_hosts` attribute. `extra_hosts` is Docker Compose / `docker run`
terminology; an operator grepping the template for `extra_hosts` will not find it.
**Fix:** Reword to reference the `host { ... ip = "host-gateway" }` block actually used in the template,
or keep `extra_hosts` only as a parenthetical analogy.
