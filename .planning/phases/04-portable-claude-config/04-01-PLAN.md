<objective>
Modify `templates/docker/main.tf` to make Claude Code config portable across all of an owner's Coder Docker workspaces. Add a per-owner shared Docker volume, mount it at a neutral path, symlink `~/.claude` and `~/.claude.json` into it from the agent startup_script, install the `claude-code` module, add the `anthropic_api_key` variable, and wrap the pattern in an inline `[REUSABLE]` comment block.

Purpose: A developer authenticates with Claude once; credentials, settings, skills, and user-scoped MCP servers persist on a per-owner volume that survives workspace delete and is reused by every subsequent workspace (CLAUDE-01..06).
Output: A single modified file — `templates/docker/main.tf` — that `terraform validate` accepts and whose source contains the exact identifiers below.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Coder admin → Terraform plan/state | `anthropic_api_key` variable value flows into plan/state; must not leak to logs |
| owner A workspace ↔ owner B workspace | Each owner's Claude credentials live on a distinct per-owner volume; cross-owner read must be impossible |
| empty root-owned volume → coder user | startup_script chown grants the coder user write access on first start via passwordless sudo |
| OAuth credentials at rest | `~/.claude.json` / `~/.claude/.credentials.json` persist on the shared Docker volume |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | per-owner config volume | mitigate | Volume name keyed on owner UUID `coder-${data.coder_workspace_owner.me.id}-claude`; a different owner gets a different volume name → distinct Docker volume, not mounted into other owners' containers (SC-4). |
| T-04-02 | Elevation of Privilege | startup_script `sudo chown` | mitigate | chown scoped narrowly to `$HOME/.claude-shared` only, never a broader path; guarded by a `stat`-based ownership check so it runs at most once (no repeated sudo). |
| T-04-03 | Tampering | nested volume mount ordering | mitigate | Home volume `volumes{}` declared before the `.claude-shared` child mount; kernel mount-namespace shadowing keeps the child subtree consistent (no accidental clobber of home contents). |
| T-04-04 | Information Disclosure | `anthropic_api_key` variable | mitigate | Variable declared `sensitive = true` so Terraform redacts it in plan/apply output and console state diffs (default `""` means nothing is set in the common OAuth path). |
| T-04-05 | Spoofing / Repudiation | OAuth credentials at rest on shared volume | accept | Credentials persist unencrypted on a host Docker volume — same trust level as the host running Coder; documented "one active workspace per owner" concurrent-write caveat (no file locking) is a known upstream race, not engineered around this milestone. |
| T-04-SC | Tampering | claude-code module / CLI install | mitigate | Module source pinned to `registry.coder.com/coder/claude-code/coder` version `5.2.0` (legitimate Coder registry, in CLAUDE.md stack). CLI itself is intentionally unpinned (D-06) — accepted churn trade-off, install scoped to the workspace agent only. |

No `high`-severity threat is left unmitigated. T-04-05 is an accepted risk consistent with the milestone's "document the caveat, don't engineer the lock" posture (REQUIREMENTS §Out of Scope).
</threat_model>
