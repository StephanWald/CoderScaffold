## Phase Goal (MVP user story)

**As a** developer using this Coder deployment, **I want to** create a workspace from the Docker template and get a working browser VS Code session and a JetBrains Gateway (IntelliJ IDEA) connection with a home directory that survives stop/start, **so that** I can do real development without losing my work between sessions.

This plan delivers the operator-facing documentation that makes Plan 01's template usable and its two known failure modes resolvable.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator → host Docker socket | The README instructs operators to set the `group_add` GID, granting the Coder server container write access to `/var/run/docker.sock` (host-equivalent privilege) |
| Operator docs → production CODER_ACCESS_URL | Connectivity guidance steers operators between the localhost/host-gateway path and a real reachable URL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03D-01 | Elevation of Privilege | Documented `group_add` socket grant | accept | Granting the Coder server access to the Docker socket is inherent to the product (it provisions containers via the socket — already true in compose.yaml from Phase 1). The README documents the minimal GID needed rather than running privileged; no new privilege beyond the existing server design. Workspace containers still get no socket access (enforced in Plan 01). |
| T-03D-02 | Information Disclosure | Operator copies a real CODER_ACCESS_URL / secrets into docs/.env | mitigate | README uses placeholder/example URLs only and points secrets at the gitignored `.env` contract (Phase 1); no real secret or token is written into README. |
| T-03D-SC | Tampering | `coder templates push` pulls pinned modules/providers | mitigate | The pushed `templates/docker/main.tf` pins all modules/providers exactly (Plan 01); the README does not instruct loosening pins or using `:latest`. No supply-chain widening introduced by the docs. |

No HIGH-severity threat introduced by documentation. The headline socket EoP is the pre-existing, accepted server design from Phase 1; this plan only documents the operator-resolved GID step.
