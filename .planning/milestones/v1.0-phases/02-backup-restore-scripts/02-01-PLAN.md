## Phase Goal

**As an** operator, **I want to** run a single non-interactive command that produces a verified
database backup, **so that** I can capture trustworthy Coder state on demand or from a scheduler
without typing a password or watching for prompts.

    Create `scripts/backup.sh` (also create the `scripts/` directory — it does not exist).
    Header line `#!/usr/bin/env bash` then `set -euo pipefail`.
    Add a short comment block at the top noting: PGPASSWORD is passed as an inline env prefix and is
    visible to local root via /proc — acceptable for this single-host deployment model (per RESEARCH

    Resolve paths from the script location, not cwd (cron runs with cwd `/`):
    `SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"` and
    `PROJECT_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"`.

    Source `.env` from `${PROJECT_ROOT}/.env` using the canonical `set -a; source "${ENV_FILE}"; set +a`
    pattern guarded by `[[ -f "${ENV_FILE}" ]]`; on absence, `echo "WARN: .env not found..." >&2` and continue.
    Do NOT use `export $(cat .env | xargs)` (breaks on spaces/`$`). Add `# shellcheck disable=SC1090`
    above the `source` line.

    After sourcing, apply defaults that mirror compose.yaml exactly:
    `POSTGRES_USER="${POSTGRES_USER:-username}"`, `POSTGRES_PASSWORD="${POSTGRES_PASSWORD:-password}"`,
    `POSTGRES_DB="${POSTGRES_DB:-coder}"`.

    Build the output path: `TIMESTAMP="$(date -u +%Y%m%d-%H%M%S)"`,
    `DUMP_FILE="${PROJECT_ROOT}/backups/coder-${TIMESTAMP}.dump"`, then `mkdir -p "${PROJECT_ROOT}/backups"`.
    `cd "${PROJECT_ROOT}"` so `docker compose` finds compose.yaml.

    Run the dump with the inline-prefix pattern: `PGPASSWORD="${POSTGRES_PASSWORD}" docker compose exec -T database pg_dump -U "${POSTGRES_USER}" -Fc --no-owner --no-acl "${POSTGRES_DB}" > "${DUMP_FILE}"`.
    The `-T` flag is mandatory (TTY corrupts the binary stream); `-Fc` is mandatory (custom format);
    `--no-owner --no-acl` are mandatory (CLAUDE.md). Service name is `database` (not `db`/`postgres`).

    Emit human-readable progress to stdout (e.g. "Starting backup: ${DUMP_FILE}") so cron logs are useful.
    As the final stdout line, print a parseable `BACKUP_FILE=${DUMP_FILE}` so a scheduler can capture the path.

<task type="auto" tdd="true">
  <name>Task 2: Add integrity verification, chmod 600, and failure exit codes</name>
  <files>scripts/backup.sh</files>
  <read_first>
    - scripts/backup.sh (the file from Task 1 — read before editing)
    - .planning/phases/02-backup-restore-scripts/02-RESEARCH.md (Pattern 4 backup verification; Security Domain ASVS V4)

  <behavior>
    - After a successful dump, the file is chmod 600 and passes a `pg_restore --list` structural check; script exits 0.
    - If the dump is zero bytes, the script removes it, prints an ERROR to stderr, and exits non-zero.
    - If the dump fails the `pg_restore --list` integrity check, the script prints an ERROR to stderr and exits non-zero.
  </behavior>
  <action>
    Extend `scripts/backup.sh` (do not rewrite Task 1 work).

    Immediately after the `pg_dump ... > "${DUMP_FILE}"` line, set restrictive permissions:
    `chmod 600 "${DUMP_FILE}"` (ASVS V4 — dump contains all user/workspace data including secrets;
    must not be world-readable).

    Size guard: `if [[ ! -s "${DUMP_FILE}" ]]; then echo "ERROR: Dump file is zero bytes: ${DUMP_FILE}" >&2; rm -f "${DUMP_FILE}"; exit 1; fi`.
    (Catches the -T-omission/corruption case where pg_dump exits 0 but produces an empty file.)

    Structural integrity guard using the dump's own TOC: pipe the file into a container-side
    `pg_restore --list /dev/stdin` and treat a non-zero exit as failure. Use the form
    `if ! PGPASSWORD="${POSTGRES_PASSWORD}" docker compose exec -T database pg_restore --list /dev/stdin < "${DUMP_FILE}" > /dev/null 2>&1; then echo "ERROR: Dump file failed integrity check: ${DUMP_FILE}" >&2; exit 1; fi`.
    Keep `-T` (binary stream) and the inline `PGPASSWORD` prefix.

    On success, print a final human-readable summary line including the dump size
    (`du -sh "${DUMP_FILE}" | cut -f1`) and keep the `BACKUP_FILE=...` parseable line as the last output.
    Because `set -e` is active, any unguarded command failure already aborts with non-zero; the explicit
    guards above add clear error messages for the two pg_dump-exits-0-but-bad cases.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| host shell → `database` container | `docker compose exec` crosses from host into the container; `PGPASSWORD` and the dump stream cross here |
| `database` container → host filesystem (`./backups/`) | binary dump stream is redirected to a host file |
| dump file at rest | the dump contains the entire Coder database (user records, tokens, workspace metadata) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | dump file at rest in `./backups/` | mitigate | `chmod 600 "${DUMP_FILE}"` immediately after write (Task 2); `backups/` already gitignored (`.gitignore` line 9) so dumps never enter version control |
| T-02-02 | Information Disclosure | `PGPASSWORD` in process listing (`/proc`) | accept | Inline env prefix on the single `exec` (not exported); visible only to local root on a single-host deployment — documented in the script header. `.pgpass` is the heavier alternative, deferred. |
| T-02-03 | Tampering / Integrity | corrupted/empty dump that pg_dump still exits 0 for (TTY corruption) | mitigate | Mandatory `-T` flag; zero-byte size guard + `pg_restore --list` structural check before exit 0 (Task 2) |
| T-02-04 | Information Disclosure | secrets leaking into stdout/cron logs | mitigate | Script never echoes `POSTGRES_PASSWORD`/`PGPASSWORD`; only the filename and size are printed. `.env` sourced via `set -a; source; set +a`, not echoed. |
| T-02-SC | Tampering | npm/pip/cargo installs | n/a | No packages installed this phase — all tooling bundled in `postgres:17` and Docker (RESEARCH Package Legitimacy Audit: N/A) |
</threat_model>
