## Phase Goal

**As an** operator, **I want to** restore a verified database backup into a fresh or existing Coder
instance with a single non-interactive command, **so that** I can recover my Coder state without
typing a password, without leaving the server stopped, and without manually dropping objects.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator-supplied argument → script | the dump-file path is untrusted input that drives a destructive operation |
| host shell → `database` container | `pg_restore` + `PGPASSWORD` cross into the container via `docker compose exec` |
| restore operation → live `coder` service | the restore drops and recreates DB objects the running Coder depends on |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Tampering | unvalidated dump-file argument (path traversal / wrong file) | mitigate | Validate `$1` is a regular file (`-f`) and non-zero (`-s`) before any destructive action (Task 1, ASVS V5) |
| T-02-06 | Denial of Service | `--clean` against the wrong/running DB leaves Coder broken or stopped | mitigate | Stop `coder` before restore; EXIT trap guarantees `docker compose start coder` even on failure (Pitfall 6); README DESTRUCTIVE warning (Task 2) |
| T-02-07 | Information Disclosure | `PGPASSWORD` in process listing (`/proc`) | accept | Inline env prefix on the single `exec`, not exported; single-host deployment; documented in script header (consistent with Plan 01 T-02-02) |
| T-02-08 | Tampering / Integrity | restore silently corrupts input (binary stream) | mitigate | Mandatory `-T`; restore from host file via stdin redirect (never the broken pipe-between-execs pattern, docker/compose #8909) |
| T-02-SC | Tampering | npm/pip/cargo installs | n/a | No packages installed — all tooling bundled in `postgres:17` + Docker (RESEARCH Package Legitimacy Audit: N/A) |
</threat_model>
