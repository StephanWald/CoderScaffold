# Quick Task 260713-m12: BBjServices Coder workspace template - Context

**Gathered:** 2026-07-13
**Status:** Ready for planning

<domain>
## Task Boundary

Create a new Coder workspace template at `templates/bbj-services/` that provisions a
Coder workspace running BasisHub **BBjServices** (https://github.com/BasisHub/docker-bbjservices)
as a live server on port 8888, alongside a full dev environment (editors + Claude Code).
Fork the existing `templates/java-fullstack/` template as the skeleton.

Upstream reference (docker-bbjservices) is a *server image*: Alpine + OpenJDK21, bakes the
BBj install at build time (`ADD BBj*.jar`, `ADD certificate.bls`, silent playback install with a
`license_server` build-arg), `EXPOSE 8888`, `CMD ["/opt/bbx/bin/bbjservices","--launchd"]`.

The critical adaptation: a Coder workspace CANNOT use `CMD bbjservices` — the container
entrypoint MUST run the Coder agent `init_script` (that is how code-server / Claude / SSH
attach). BBjServices must therefore be launched *by* the agent `startup_script` as a
background service, with the agent staying in the foreground.

</domain>

<decisions>
## Implementation Decisions (LOCKED — do not revisit)

### Scope
- Dev + live server. Provide code-server + JetBrains Gateway (IntelliJ) + Claude Code
  (keep everything java-fullstack already wires up), AND run BBjServices on 8888.

### BBj install timing — bake at build from a HOST build-context folder
- The BBj installer JAR (`BBj*.jar`) and `certificate.bls` are OPERATOR-SUPPLIED and live
  ONLY in a folder on the Docker host — they must NEVER be committed to git.
- `playback.properties` (the silent-install answer file) IS version-controlled in the template.
- The image build context is that host folder, which must be readable by the `coder`
  container (the provisioner that runs the build) — so it is bind-mounted into the `coder`
  service in compose.yaml.

### 8888 exposure — Coder app / subdomain
- Register a `coder_app "bbjservices"` → `url = "http://localhost:8888"`, `subdomain = true`,
  `share = "owner"`, with a healthcheck. Dashboard button + wildcard-subdomain access.

### JDK — Adoptium 21 (default) + 25
- `coder_parameter "jdk"` offers `adoptium-21` (default) and `adoptium-25`, reusing
  java-fullstack's Adoptium build install. The `adoptium-25` option description MUST warn
  that it requires a BBj build that supports JDK 25 (upstream ships only 21).

### License server
- Upstream's `license_server` build-arg (host:port) → template variable `bbj_license_server`
  → passed as the `LICENSE_SERVER` Docker build-arg → `sed`'d into `playback.properties`
  replacing `LICENSE_SERVER_PLACEHOLDER`, exactly as upstream does.

### Claude's Discretion
- Exact startup_script mechanism to background bbjservices (nohup/&/setsid) and any
  readiness wait, provided the agent process stays foreground and the server keeps running.
- Whether EXPOSE 8888 is kept in the Dockerfile (harmless; coder_app only needs the agent
  to reach localhost:8888 inside the container).
- Image name / triggers wiring, as long as JDK, LICENSE_SERVER, and the JAR checksum all
  force a rebuild when changed.

</decisions>

<specifics>
## Specific Ideas — concrete pieces to produce

1. **`templates/bbj-services/Dockerfile`** — Ubuntu base (`codercom/enterprise-base:ubuntu`,
   NOT upstream's Alpine — the Coder editor modules expect Debian/Ubuntu). Reuse
   java-fullstack's Adoptium JDK 21/25 build-time install (build-arg `JDK`) + the Node.js,
   MemPalace, and SSH/known_hosts layers. Then the upstream BBj sequence:
   - `ADD playback.properties /tmp/playback.properties`
   - `ADD BBj*.jar /tmp/BBj.jar`
   - `ADD certificate.bls /tmp/certificate.bls`
   - `ARG LICENSE_SERVER` + `RUN sed -i "s/LICENSE_SERVER_PLACEHOLDER/${LICENSE_SERVER}/g" /tmp/playback.properties`
   - `RUN java -Djdk.lang.Process.launchMechanism=vfork -jar /tmp/BBj.jar -p /tmp/playback.properties`
   - upstream `BBj.properties` seds (vfork launch arg, defaultAdminUserWarning=false, webUser=root)
   - upstream `umask 0` + cleanup (cache/logs/foreign libnative/documentation, aws/BBjTextSearch jars)
   - NO `CMD bbjservices`.
   - `USER coder` at the end.
   Reference existing files verbatim for the reused layers:
   - `templates/java-fullstack/Dockerfile` (JDK/Node/MemPalace/SSH layers to copy)
   - Upstream Dockerfile BBj steps captured in this context.

2. **`templates/bbj-services/main.tf`** — fork `templates/java-fullstack/main.tf`. Changes:

     e.g. `/mnt/bbj-assets`); `dockerfile` must resolve within that context — since the
     Dockerfile + playback.properties are version-controlled in the template but the JAR/cert
     live in the host folder, DOCUMENT that the operator ships `Dockerfile` + `playback.properties`
     into the host context folder (README covers this), and point context at that folder.
   - Add `LICENSE_SERVER = var.bbj_license_server` to `build_args`; add the JAR checksum and
     `bbj_license_server` to `triggers` so a new JAR or license server forces a rebuild.
   - `coder_agent.main.startup_script`: after the existing setup, launch BBjServices in the
     background (nohup/&) so it keeps running while the agent stays foreground. Idempotent and
     non-fatal (warn-and-continue), matching the template's existing WR-03 style.
   - New `coder_app "bbjservices"` as decided above (subdomain=true, share=owner, healthcheck).
   - New variables: `bbj_context_path` (build context / host asset folder) and

6. **`templates/bbj-services/README.md`** — operator setup guide:
   - Prereqs: a running BLS license server, a downloaded `BBj*.jar`, and `certificate.bls`.
   - Drop `BBj*.jar` + `certificate.bls` into the host asset folder (`BBJ_ASSETS_PATH`).
   - Ship `Dockerfile` + `playback.properties` into that same folder (it is the build context).
   - Set `BBJ_ASSETS_PATH` + `BBJ_LICENSE_SERVER` in `.env`, restart compose.
   - Push the template (`coder templates push`) and create a workspace; reach BBjServices via
     the Coder app on 8888.
   - Document the three FLAGS below.

## FLAGS to document (not block on)
- `coder_app subdomain=true` requires `CODER_WILDCARD_ACCESS_URL` + an external proxy routing
  `*.<apps-domain>`. On the committed default (`CODER_ACCESS_URL=127.0.0.1`, no wildcard) the
  8888 button will not route — note the path-based (`subdomain=false`) fallback.
- JDK 25 requires a BBj build that supports JDK 25 (upstream ships only 21).
- `/opt/bbx` is baked into the image, NOT persisted — config changes there reset on an image
  rebuild. `/home/coder` IS persistent (Docker volume). Note this; an `/opt/bbx` volume can be
  added later if persistence is needed.
