# Quick Task 260717-73u: Rebase bbj-ls-dev on bbj-services Pattern Summary

`bbj-ls-dev` now bakes a complete, coder-owned BBj install (`/opt/bbx`) via a three-stage Dockerfile identical in structure to `bbj-dev`, and its `main.tf` consumes the shared `./modules/bbj-base` module instead of an inline house pattern — while six `moved` blocks preserve existing workspaces' persistent home volumes.

## What Was Done

### Task 1: Multi-stage Dockerfile + combinations.json (commit 121ab63)
- Rewrote `templates/bbj-ls-dev/Dockerfile` as `base` -> `bbjinstall` -> `final`, following `bbj-dev/Dockerfile`'s structure.
- `base`: kept Node LTS, the dual JDK 17+21 install (always), Maven pinned, profile.d exports, gh CLI, the git.basis.cloud:10022 keyscan (plus github/gitlab/bitbucket). Added a JDK-selection block: `adoptium-21` symlinks `/opt/java/default` to the already-baked 21; `adoptium-25` additionally downloads Temurin 25 and defaults to it; any other value fails the build. Added the MemPalace venv block (verbatim from bbj-dev) — required because `module.base`'s startup script registers the mempalace MCP server unconditionally.
- `bbjinstall`: copied verbatim from bbj-dev (playback.properties + BBJ_JAR_NAME + certificate.bls, LICENSE_SERVER sed with empty-warning, vfork install, BBj.properties post-install sed, logs/cache/documentation trim keeping the full lib set).
- `final`: `COPY --from=bbjinstall --chown=coder:coder /opt/bbx /opt/bbx`, `EXPOSE 8888`, `USER coder` last.
- `combinations.json` (new file): `bbj-26.01-jdk25` (production-obf, JDK 25) first, `bbj-trunk-nonobf-jdk21` second — matches the plan's required production-default ordering.
